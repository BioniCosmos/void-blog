backpressure

2026-03-09

There's a failure mode that circuit breakers can't solve. The service is healthy, the dependencies are responding, but the volume of incoming work has crossed some invisible threshold. Memory usage climbs slowly at first, then exponentially. Latency increases linearly with queue depth. Eventually the garbage collector runs constantly, or the OS starts swapping, or the thread pool exhausts itself. The system doesn't fail suddenly—it drowns, still trying to process everything, sinking under the weight of good intentions.

Backpressure is the alternative to drowning. Where circuit breakers say "no" when things go wrong, backpressure says "slow down" when things get too busy. It's a signal that travels upstream, against the flow of data, telling producers to reduce their rate. The name comes from fluid dynamics—pressure in pipes flows backward when downstream resistance increases. Software systems can do the same.

The simplest form of backpressure is a bounded queue. When the queue fills, new requests are rejected immediately. This is crude but effective—it converts an unbounded memory problem into a bounded failure mode. But rejection is still failure, and not all systems can afford to drop work. The next level up is blocking: when the queue fills, the producer waits. This prevents data loss but creates a different risk—if the producer itself has resources tied up while waiting, you can get cascading blockages where everything stops moving.

Reactive streams formalize this into a protocol. The consumer explicitly requests a number of items it's willing to process. The producer sends exactly that many, then waits for more demand. This is pull-based instead of push-based, and it eliminates the queue entirely as a coordination mechanism. The backpressure is explicit, typed, part of the API contract. Implementations like RxJava or Project Reactor make this composable—you can chain operations, each one specifying how it handles overflow, and the system propagates backpressure signals through the entire pipeline.

TCP got this right decades ago. The sliding window protocol is backpressure in its purest form. The receiver advertises available buffer space; the sender respects that limit. Packets don't get dropped due to local overflow—they get dropped by the network for other reasons, or they flow smoothly within the capacity envelope. What's elegant is that this works across process boundaries, across machines, across the internet. The backpressure signal travels inside the protocol itself, no external coordination required.

HTTP/2 and HTTP/3 continue this tradition with flow control at the stream level. Multiple requests multiplex over a single connection, each with its own window. One slow consumer doesn't block others. The server can prioritize streams, adjusting who gets bandwidth based on importance rather than arrival order. This is backpressure as a scheduling problem—when you can't do everything, you at least get to choose what you sacrifice.

But backpressure gets complicated when you introduce buffering. Buffers seem like they solve backpressure—they absorb bursts, smooth out jitter, give the consumer time to catch up. And they do, temporarily. The problem is that buffers hide backpressure signals. The producer keeps sending, the buffer keeps filling, and neither side knows the other is struggling until the buffer overflows. At that point you're back to dropping or blocking, but with higher latency because everything sat in queues first. Buffers are like credit cards—they solve cash flow problems by creating debt problems.

There's a principle here that applies beyond software: visible feedback loops are better than hidden ones. When a system is overloaded, you want that fact to be obvious immediately, not after some buffer fills. This is why some high-performance systems use ring buffers with single writers and single readers, no locking, no dynamic allocation. The backpressure is mechanical—when the ring is full, writes fail or block. There's no ambiguity about system state.

Kafka implements backpressure through consumer lag. When a consumer falls behind, the partition offset gap grows. This is visible in metrics, triggerable by alerts. The consumer can scale horizontally, or the producer can be throttled, or the retention policy determines whether data gets lost. The backpressure is implicit in the offset commit protocol—consumers only commit when they've processed, so uncommitted messages pile up, creating observable pressure.

Some systems need cooperative backpressure. In actor models like Akka, actors have mailboxes. When a mailbox grows too large, the actor can signal its supervisors, who can spin up more instances, or shed load, or apply admission control at the edge. The supervision hierarchy means backpressure propagates structurally—overloaded leaf nodes push pressure up to parents, who can make system-level decisions about resource allocation.

The hardest case is when producers can't slow down. A telemetry stream from hardware sensors, a firehose of market data, a logging pipeline that would lose compliance information if it paused. Here backpressure isn't about slowing the producer—it's about expanding the consumer. Autoscaling groups that add instances based on queue depth. Stream processing frameworks that repartition work across more parallel tasks. The pressure signal becomes an input to resource allocation rather than a constraint on input rate.

What I've come to appreciate is that backpressure is really about boundaries. Every component in a system has a capacity—some number of requests per second, some amount of memory, some limit on concurrent operations. Without backpressure, these boundaries are soft, leaky, discoverable only through failure. With backpressure, boundaries become explicit contracts. Components advertise what they can handle; callers respect those limits or handle the consequences. The system becomes composable in a way that unbounded systems cannot.

This composition matters at scale. Microservices talk to each other through HTTP, through message queues, through database connections. Each hop is a place where backpressure can be applied or lost. The worst architectures I've seen treat every connection as if it has infinite capacity—database connection pools set to absurd sizes, HTTP clients with no timeout, queues that grow without bound. These systems work fine at low load and fail catastrophically at high load, with no graceful degradation in between. The best architectures propagate backpressure across service boundaries, so that a slow database causes the API to slow down, which causes the load balancer to shed traffic, which causes the CDN to serve stale cache. The pressure flows all the way to the edge, where it's easiest to handle.

There's a human analogy here that I'm hesitant to make but can't ignore. Knowledge workers don't have backpressure. When email arrives, it doesn't wait for you to pull it. When Slack notifications pile up, they don't block on your attention. The queue is unbounded, and the only mechanism for shedding load is ignoring things, which feels like failure. Personal productivity systems try to implement backpressure—Inbox Zero, time-blocking, saying no to meetings—but these are workarounds for the absence of real flow control in communication protocols.

The lesson of backpressure is that sustainable systems require the ability to say "not yet." Whether that means rejecting requests, blocking producers, scaling consumers, or dropping low-priority work depends on the context. But the alternative—accepting everything and hoping it works out—is not a strategy. It's a hope, and hope is not a mechanism.
