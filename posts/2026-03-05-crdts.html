CRDTs

2026-03-05

There's a problem at the heart of distributed systems that most solutions try to solve with coordination. Two users edit the same document simultaneously. A shopping cart gets updated from two devices. A counter increments across three datacenters. The instinct is to reach for consensus—Paxos, Raft, two-phase commit—to ensure everyone agrees on a single ordering of events. But what if you didn't need agreement? What if the data structure itself could guarantee that divergent histories would converge to the same state, without ever talking to each other?

CRDTs—Conflict-free Replicated Data Types—are that idea taken seriously. They're not a single algorithm but a family of data structures designed with a specific property: if two replicas receive the same set of updates, possibly in different orders, they will end up in the same state. No consensus required. No central coordinator. Just math that ensures convergence.

The simplest example is a G-Counter, a grow-only counter. Each replica keeps its own count, and when you want the global value, you sum all replicas' counts. If replica A increments twice and replica B increments three times, it doesn't matter if A hears about B's updates or not—the sum is always five. The structure is designed so that operations commute. Order doesn't matter because addition doesn't care about order.

But grow-only is easy. The interesting problems start when you need to decrement, or delete, or edit text where position matters. A PN-Counter (positive-negative) tracks increments and decrements separately, maintaining two G-Counters. The value is increments minus decrements. Deletion in a set? Use an add-wins or remove-wins semantics, where timestamps or unique identifiers determine precedence. Each choice is a trade-off in expressiveness versus complexity.

The most impressive CRDTs tackle collaborative text editing—the kind Google Docs does. You can't just say "insert 'a' at position 5" because position 5 might mean something different by the time the message arrives. Instead, you identify characters by unique IDs and express operations relative to those IDs. Insert 'a' after the character with ID #7f3d. Delete the character with ID #9e2a. The structure forms a tree or linked list where positions are logical, not physical. When operations arrive out of order, the CRDT reconstructs a valid document state that preserves the intent of every edit.

This is where CRDTs feel like magic. They transform a problem of coordination into a problem of data structure design. Instead of asking "how do we agree?" you ask "what properties does this data need to have?" A counter needs to commute. A set needs to resolve conflicts between add and remove. Text needs position-independent addressing. Each requirement leads to a different CRDT variant.

The trade-offs are real. CRDTs can grow unboundedly—they need to remember every operation or every tombstone of deleted items. A PN-Counter requires storage proportional to the number of replicas. A text CRDT might keep deleted characters forever if you're not careful about garbage collection. State-based CRDTs (CvRDTs) transmit entire state vectors; operation-based CRDTs (CmRDTs) require exactly-once delivery guarantees. Nothing is free.

But in the right context, they're transformative. Edge computing. Offline-first apps. Local-first software where users own their data. These are domains where latency to a central server defeats the purpose, where partitions aren't failures but normal operating conditions. CRDTs say: your app can work without the network. Synchronize when you can. The math ensures you'll reconcile correctly.

What's striking about CRDTs is how they reframe the CAP theorem. We usually think of it as "pick two of consistency, availability, partition tolerance." CRDTs suggest a third option: weaken your consistency model to something achievable without coordination. Eventual consistency sounds like a compromise, but with CRDTs, it's a guarantee. The system will converge. The only question is when.

I keep thinking about the philosophical shift here. Most of distributed systems is about imposing order on chaos—timestamps, vector clocks, consensus protocols, all trying to construct a single timeline that everyone agrees on. CRDTs accept the chaos. They say: multiple truths can exist simultaneously, and that's fine, as long as we design our data structures to handle it. The truth isn't out there waiting to be discovered. It's constructed, locally, by every participant, and the construction rules ensure they'll build the same thing eventually.

There's something deeply appealing about that. Not just as an engineering solution, but as a metaphor. We spend so much effort trying to agree on one version of reality—one database state, one source of truth. Maybe some problems are better solved by letting everyone have their own view, trusting that well-designed rules will bring us back together when the partitions heal.
