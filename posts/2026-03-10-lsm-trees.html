lsm trees

2026-03-10

There's a fundamental tradeoff in storage engines that most people don't notice until it hurts them. B-trees dominate the world — they're in PostgreSQL, MySQL, SQLite, virtually every relational database you've touched. They've been the default answer to "how do I store data on disk" for decades. But there's a whole class of workloads where B-trees are exactly wrong, and the alternative is both elegant and surprisingly simple.

The problem is write amplification. In a B-tree, every insert or update has to find the right leaf node, read it from disk, modify it, and write it back. If you're unlucky and the node splits, you cascade changes up the tree. A single logical write becomes multiple physical writes. On spinning disks this was annoying. On SSDs, which wear out with each write cycle, it's actively expensive. But the real pain comes at scale, when your write-heavy workload saturates the disk's IOPS budget just keeping the tree balanced.

Log-structured merge trees flip the problem inside out. Instead of updating data in place, you don't update at all. Every write — insert, update, delete — gets appended to a sequential log. No seeking, no random I/O, no in-place modification. The writes are batched in memory, sorted, and periodically flushed to disk as immutable files. These files are called SSTables: sorted string tables. Once written, they never change.

The immutability is the key insight. If files never change, you don't need complex locking for concurrency. You don't need to worry about half-written pages if the system crashes. You can cache aggressively because the data isn't going to change under you. And you can serve reads from memory while compactions happen in the background, completely decoupled from the write path.

But immutability creates a new problem: if you keep appending updates, how do you find the current value of a key? The answer is layering. Each flush creates a new SSTable at level 0. When you read, you check the in-memory memtable first. If it's not there, you check the most recent SSTable, then the next, and so on. A key might exist in multiple files with different values — the first one you find is the current truth.

This means reads get slower as you accumulate more files. You're trading write performance for read performance, which seems backwards until you remember the target workload: write-heavy, read-light, or reads that can be served from cache. For many modern applications — metrics ingestion, logging, event streaming — this is exactly the right tradeoff.

To keep reads from degrading completely, LSM trees periodically merge files. This is the "merge" in the name. Small files at level 0 get compacted into larger files at level 1, which eventually get compacted into even larger files at level 2, and so on. Each compaction reads several files, merges them by key, and writes out new files with duplicates eliminated. Old values are garbage collected. Deleted keys (written as tombstones) are finally purged.

Compactions are where the complexity hides. Do them too aggressively and you burn I/O and CPU, defeating the purpose of the LSM tree. Do them too lazily and reads suffer as you search through dozens of files. The tuning knobs here — size ratios between levels, compaction triggers, whether to prioritize write throughput or read latency — are the difference between a storage engine that flies and one that mysteriously stalls.

RocksDB, Facebook's fork of LevelDB, exposes so many of these knobs that configuration becomes an art. You can choose between leveled compaction, which optimizes for read performance by keeping the number of files small, and tiered compaction, which optimizes for write throughput by delaying merges. You can pin hot data in memory, filter keys with Bloom filters to avoid unnecessary disk reads, and compress data at different levels with different algorithms.

The delete operation in an LSM tree is particularly clever. Since you can't modify files, you can't remove a key. Instead you write a tombstone — a special marker that says "this key is deleted." Future reads see the tombstone and return nothing. Compactions eventually strip out both the tombstone and the data it masks. But until then, deletes are as fast as writes, which is to say: very fast.

This design pattern — accept the write, defer the work, clean up asynchronously — shows up everywhere once you start looking. It's in message queues, in garbage collectors, in event sourcing architectures. The LSM tree just applies it to persistent storage. The insight is that sequential I/O is so much faster than random I/O that you can afford to do more total work if you keep it sequential.

Cassandra and ScyllaDB use LSM trees under the hood. So does CockroachDB. Even some graph databases have switched from B-trees to LSMs for their write-heavy workloads. The pattern has become the default choice for anything that needs to absorb a firehose of data and query it eventually. Not immediately — eventually. The LSM tree makes you think explicitly about the difference.

The lesson isn't that B-trees are obsolete. For read-heavy transactional workloads, for range scans, for anything where consistency and predictable latency matter more than raw write throughput, B-trees are still king. The lesson is that storage engine design is about choosing your pain. B-trees make writes expensive and reads cheap. LSM trees make writes cheap and reads eventually consistent. Both are correct. The trick is knowing which one you're building.
